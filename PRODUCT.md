# Product

## Register

product

## Users

Developers who own local repositories, GitHub access, agents, and session cleanup; invited product managers who request scoped changes without terminal or direct repository access.

## Product Purpose

PairDock lets a developer configure one local agent for a project and share that ready project with an invited PM. The PM can open multiple isolated feature or fix sessions, converse with the developer's agent, inspect the live preview, validate the result, and request a draft pull request. The developer retains control of source access, model, reasoning, execution, and cleanup.

## Brand Personality

Direct, technical, collaborative. Interface copy should explain concrete system state and the next available action without marketing filler.

## Anti-references

- Placeholder navigation, inactive controls, fake dashboards, or views without implemented behavior.
- Decorative SaaS metrics that do not help complete the current workflow.
- Terminal access or repository controls exposed to PM users.
- UI implying production, CI, policy, or multi-model capabilities outside the MVP.

## Design Principles

- Show only implemented MVP actions and state.
- Keep developer and PM responsibilities visibly separate.
- Make readiness, authorization, agent availability, and failure states explicit.
- Keep the primary workflow short: connect, configure, share, prompt, validate, draft PR.
- Prefer server-derived truth over client-selected provider or permission state.

## Accessibility & Inclusion

Use semantic controls, visible keyboard focus, readable contrast, explicit labels, and text alongside status colors. Preserve usable layouts at mobile and desktop widths.
